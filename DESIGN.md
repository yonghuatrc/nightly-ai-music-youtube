## Overview

Redesign the existing nightly AI music pipeline (2 songs/night via MiniMax API → Telegram) into a YouTube channel that auto-generates quality music daily with visualizers. Target: 500 subscribers in 1 month. Transparent about AI origin but focused on quality output.

## What Makes This Cool

AI-generated music that's actually good — uploaded daily to YouTube with visualizers, showing what's possible. Full YouTube experience: thumbnails, descriptions, playlists, channel branding. The pitch: "AI songs can be good. Here's proof, every day."

## Premises

1. ✅ The existing MiniMax API + Telegram pipeline works but YouTube needs entirely new infrastructure (visualizer generation + YouTube Data API v3 upload + channel management)
2. ✅ Start with static visualizer (album art + waveform) → iterate toward animated visualizers → eventually full AI music videos. Ship fast, polish later.
3. ✅ 500 subscribers in 1 month requires BOTH consistent daily uploads AND distribution strategy
4. ✅ This needs multiple agents — minimum: music gen, visualizer, YouTube uploader, scheduling coordinator

## Approaches Considered

### Approach A: Minimal Viable (Phase 1 — NOW)

Extend existing pipeline with static visualizer + YouTube upload. 2 songs/night. Use existing MiniMax music gen, add FFmpeg waveform visualizer, YouTube Data API v3 upload with scheduling. Telegram stays. Ships in days.

**Effort:** M (3-5 days) | **Risk:** Low

### Approach B: Multi-Agent YouTube Machine (Phase 2 — AFTER validation)

Coordinator agent → sub-agents: Music Gen (N songs configurable), Visualizer (static → animated), YouTube Uploader (SEO, playlists), Growth Agent (shorts, cross-posts). Config-driven YAML.

**Effort:** L (2-3 weeks) | **Risk:** Low-Med

### Approach C: Content Factory (deferred)

Same as B plus Shorts, community posts, LLM-driven SEO. Maximum reach per song.

## Recommendation: A first, then B

Ship Phase 1 to get content on YouTube ASAP. Validate workflow. Learn what resonates. Then build the multi-agent architecture.

## Pipeline Architecture (Phase 1)

```
CRON 2am
  │
  ▼
nightly_music.py --date YYYY-MM-DD
  │
  ├── Load config (nightly-music.yaml)
  ├── Create daily dir (YYYY-MM-DD/)
  ├── Fetch trending songs (fetch_trending.py)
  │     └── Dedup against last 7 days (check-duplicate.py)
  ├── Generate songs (MiniMax API, 2 songs)
  │     └── Save MP3 + TXT to daily dir
  ├── Generate visualizers (nightly_visualizer.py)    ← NEW
  │     └── FFmpeg waveform + album art → MP4
  ├── Upload to YouTube (nightly_uploader.py)          ← NEW
  │     └── YouTube Data API v3, scheduled 6pm
  ├── Sync to D-drive
  ├── Log to song-log.json (monthly rotation)
  └── Telegram delivery (sendMediaGroup, summary + links)
```

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| MiniMax API downtime at 2am | Songs missed | Retry once; skip song on second failure |
| YouTube OAuth token expires | No upload | Auto-refresh handles this; fallback: Telegram alert |
| YouTube API quota exceeded | No upload | 2 songs = 3,400/10,000 units (34%), safe |
| YouTube copyright claim | Channel risk | Channel is transparent about AI origin |
| D-drive not mounted | Sync fails | Non-fatal; warn and continue |
| Log file corruption | Data loss | Monthly rotation isolates damage |
| Duplicate YouTube upload | Wasted quota | Idempotent song log prevents re-upload |
